# 향후 개발 방침 (2026-06-14 정리)

## 1. enrich 기능(용법·용량 자동 보완)은 폐지한다
- 웹 검색 기반 enrich(api/enrich.js)는 삭제한다.
- 이유: 용법·용량은 제품 포장이나 첨부문서에 적혀 있어, 웹 검색
  비용을 들여 가져올 가치가 낮다(테스트 중 몇 번 만에 약 $5 소비).
  공식 정보를 찾지 못하는 제품도 있다.
- 용법·용량이 필요하면 포장의 해당 면을 촬영해 OCR로 읽는 방식으로 한다.
- 가격 비교 등 향후 웹 검색 계열 기능에서는 enrich에서 얻은 노하우
  (웹 검색 + 캐시 + 비용 대책)를 재사용한다.

## 2. 신규 기능 "복약 점검(상담 기능)"
사용자가 등록한 약·영양제 전체를 한 번에 모아서 점검하는 기능.

**위치 설정**
- 의료 판단은 하지 않는다. "알아차림"과 "상담의 계기"를 제공하는 보조 역할.
- 웹 검색을 쓰지 않음(등록된 데이터를 AI에 넘기기만 함)으로 저비용.
  버튼을 눌렀을 때만 실행한다.

**입력(처방약과 영양제를 구분해서 전달)**
- 병원에서 처방받은 약 = 반드시 먹는 "기반"
- 영양제 = 선택적으로 먹는 "추가"
- 앱은 이미 "약/영양제" 종류 구분을 가지고 있으므로 이를 활용한다.

**출력(한 번에 모아서)**
- 영양제 성분 중복
- 같은 목적의 영양제 겹침
- 처방약과 영양제 사이에서 일반적으로 주의되는 조합
- 복용 타이밍의 일반적인 기준

**답변 규칙(가장 중요·안전 설계)**
- 불안을 부추기지 않는다. 모든 것을 "위험"으로 표현하지 않는다.
- "확인하는 게 좋은 점"과 "일반적으로 특별히 문제가 알려지지 않은 점"을
  차분하게 나누어 전달한다.
- "절대", "위험", "끊어야 한다" 같은 단정 표현을 쓰지 않는다.
  "일반적으로 ○○라고 합니다" 정도로만 말한다.
- 사용자의 체질·병력·정확한 처방은 알 수 없다는 전제로 쓴다.
- 반드시 마지막에 의사·약사와의 상담을 권한다. 앱의 목록을 보여주면
  전달하기 쉽다는 점도 덧붙인다.

**UI**
- 버튼 이름은 "복약 점검 / 같이 먹어도 되는지 확인" 계열(친근하게).
- 버튼 아래나 결과 상단에 "중복이나 복용 타이밍도 함께 확인합니다" 보충.

**배경(왜 이 설계인가)**
- 개발자 본인의 경험: 영양제를 여러 개 먹고 있었으나, 처방약 시작 후
  의사로부터 하나만 "끊는 게 좋다"는 말을 듣고, 불안해서 전부 끊어버렸다.
- 정보가 없으면 "전부 끊기" 같은 극단적 행동으로 치닫기 쉽다.
- 이 기능은 극단적 판단을 막고, 전문가 상담을 뒷받침하는 것이 목적이다.
